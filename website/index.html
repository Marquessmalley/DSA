Time Complexity (Big-O)

Zero-Math Version

This is one of the most important lessons in all of DSA.
If this clicks, everything else becomes easier.

1. What Problem Does Time Complexity Solve?

Imagine you write a program and ask:

"Will this still work if my input becomes very large?"

Time complexity helps answer:

  • Will it be fast enough?
  • How does runtime grow as input grows?
‚ö†Ô∏è

We are not measuring seconds

We are measuring growth

2. The Core Idea (In One Sentence)

Time complexity describes how the number of operations grows with input size.

Input size is usually called n.

3. Let's Start with the Simplest Example

Example: One Loop
numbers = [2, 4, 6, 8, 10]

for num in numbers:
    print(num)

Let's analyze:

If list size = 5 ‚Üí loop runs 5 times
If list size = 100 ‚Üí loop runs 100 times
If list size = n ‚Üí loop runs n times
üëâ O(n) "order of n"

Meaning:

  • Runtime grows linearly
  • Double the input ‚Üí double the work

4. Why We Ignore Exact Numbers

for num in numbers:
    print(num)      # 1 operation
    print(num * 2)  # another operation

Runs: 2 operations per element = 2n total

We still say: O(n)

Constants (like 2, 5, 100) don't matter for growth

Growth trend matters

5. Constant Time: O(1)

numbers = [10, 20, 30]

print(numbers[0])

No matter:

  • List size = 3
  • List size = 1 million

This always takes the same time.

üëâ O(1) constant time

6. Two Loops = O(n¬≤)

numbers = [1, 2, 3]

for i in numbers:
    for j in numbers:
        print(i, j)
Outer loop runs n times
Inner loop runs n times for each outer loop
Total operations ‚âà n √ó n = n¬≤
üëâ O(n¬≤) quadratic time
n = 10 ‚Üí 100 operations
n = 1,000 ‚Üí 1,000,000 operations üò¨

7. Important Clarification

Common Beginner Confusion

This is NOT O(n¬≤):
for num in numbers:
    print(num)

for num in numbers:
    print(num)

This is: n + n = 2n ‚Üí Still O(n)

Pattern Result
Nested loops Multiply
Separate loops Add

8. The Three Most Important Big-O Types

O(1) Constant Access by index
O(n) Linear Single loop
O(n¬≤) Quadratic Nested loops

You'll see these everywhere.

9. How to "Read" Time Complexity

A skill you must build

Look at code and ask:

  1. How many loops?
  2. Are they nested?
  3. What grows with input size?

That's it.

10. Tiny Exercises

üëâ Time complexity?

11. One Thing to NOT Worry About Yet

  • ‚ùå Exact operation counting
  • ‚ùå Mathematical proofs
  • ‚ùå Space complexity (we'll do it later)

Right now, intuition > precision.

03

Arrays (Python Lists)

Your First Data Structure

This is where DSA stops being abstract and starts feeling real.

1. What Is an Array?

An array is a collection of elements stored in contiguous memory locations.

In Python, we use lists, but conceptually they behave like arrays. 

numbers = [10, 20, 30, 40]
Key Characteristics
  • Ordered
  • Indexed (starts at 0)
  • Stores elements of the same conceptual type
  • Fast access by index

2. Why Arrays Exist

Arrays solve a simple problem:

"How do I store many values and access them efficiently?"

Without arrays, you'd need separate variables:

a = 10
b = 20
c = 30
d = 40

Arrays give you:

  • ‚Üí Structure
  • ‚Üí Predictability
  • ‚Üí Performance

3. Indexing: The Superpower of Arrays (O(1))

numbers = [10, 20, 30, 40]

print(numbers[0])  # 10
print(numbers[2])  # 30

No matter how big the array is:

  • Accessing numbers[i] takes the same time
üëâ O(1) index access
üì¨
Mental Model

Imagine a row of mailboxes with numbers on them.
You don't open every mailbox ‚Äî you go directly to the one you want.

4. Traversing an Array (O(n))

for num in numbers:
    print(num)
4 elements ‚Üí 4 operations
n elements ‚Üí n operations
üëâ O(n) traversal

This should feel intuitive after Lesson 2.

5. Updating Values by Index (Still O(1))

numbers[1] = 99

Why?

  • Python already knows where index 1 lives
  • No shifting
  • No looping
üëâ O(1) update by index

6. Searching in an Unsorted Array (O(n))

numbers = [7, 14, 21, 28]
target = 21

for n in numbers:
    if n == target:
        print("Found")

Worst case:

  • Target is last
  • Or not present

You must check every element.

üëâ O(n) unsorted search
üëâ This is why algorithms matter, not just data structures.

7. Appending to an Array (Almost O(1))

numbers.append(50)

Most of the time:

  • Adds element to the end
  • Very fast

Occasionally:

  • Python needs more memory
  • Copies elements
üëâ O(1)* amortized
‚ö†Ô∏è

Don't worry about "amortized" yet

Just remember append is usually fast

8. Inserting in the Middle (This Is Expensive)

numbers.insert(1, 15)

What happens internally:

  • Elements after index 1 shift right
  • Many assignments occur

If array has n elements ‚Üí up to n shifts

üëâ O(n) middle insertion
üëâ This is one of the biggest weaknesses of arrays.

9. Removing Elements (Also O(n))

numbers.pop(1)

Same problem:

  • Elements must shift left
  • Cost grows with size
üëâ O(n) removal (not from end)

10. Summary Table (Very Important)

Operation Time Complexity
Access by index O(1)
Update by index O(1)
Traverse O(n)
Search (unsorted) O(n)
Append (end) O(1)*
Insert / delete middle O(n)

* Amortized

11. Tiny Exercises (Think, Don't Code)

1

Why is index access O(1) but search O(n)?

Index gives direct position (like a house address). Search must check each element until found.
2

If an array doubles in size, what happens to traversal time?

Traversal time also doubles ‚Äî O(n) means linear relationship with size.
3

Why might arrays be a bad choice for frequent insertions?

Each insertion requires shifting elements ‚Äî O(n) cost adds up quickly with frequent inserts.

12. Key Takeaway (Lock This In)

Arrays are fast for reading, slow for shifting.

If you remember only one sentence from this lesson, make it that.
